### US-002: Create config-driven CLI entrypoint
Status: in_progress
Depends on: US-001

Description:
As a developer, I want a single executable module entrypoint so I can run the pipeline in miso without manually stitching together export, train, and evaluation calls.

Acceptance Criteria:
- [ ] Add a CLI or module entrypoint that reuses the existing codebase internals and is callable as python -m alarm_hgt.train --config configs/alarm_hgt.yaml --run-mode smoke.
- [ ] The entrypoint must create outputs/checkpoints/ and outputs/results/ when absent and must keep smoke-run artifact naming deterministic.
- [ ] Example: running the smoke command on a valid CPU-friendly config logs export, train, validation, test, artifact save, and exits with code 0.
- [ ] Negative case: an invalid config path, invalid run-mode value, or unwritable output directory causes a non-zero exit and does not print a false success message.
- [ ] Add tests that prove the entrypoint can build runtime objects from the YAML file path without relying on hidden hardcoded paths.
